# MADL 语言规范

## Version 0.1.0

---

## 1. 引言

### 1.1 目的

MADL（Markdown Agent Description Language）是一种嵌入于 Markdown 文档的结构化执行语言，用于定义可执行 Flow、Agent 协调逻辑、工具调用、宿主交互以及 AI 调用上下文。

MADL 的设计目标包括：

1. **文档与程序共存**：源文件既是可读文档，也是可执行描述。
2. **语义确定**：控制流、数据流、并发、等待与恢复都必须具有稳定且可实现的语义。
3. **运行时友好**：语言应支持长生命周期任务、宿主委托并发、异步恢复、人工参与流程及 AI 工具循环。
4. **静态可分析**：解析器、检查器和运行时应能够在执行前发现大部分结构错误、作用域错误与非法引用。
5. **组合优先**：一个文档定义一个 Flow，通过统一的调用、绑定与宿主协作机制组合更复杂行为。

### 1.2 范围

本规范定义：

* 文档模型与头部定义
* 指令识别与块级结构
* 表达式、值模型与绑定模型
* `const/value` 声明、赋值与追加语义
* 控制流、迭代、并行
* 宿主交互语句
* AI 单次调用、Agent 上下文、AI 工具以及 Agent 自动工具循环
* 错误条件与一致性要求

本规范不定义：

* 具体 Markdown 解析器实现
* 具体表达式引擎实现
* 具体宿主 API 形态
* 具体模型厂商接口协议
* 具体持久化编码格式

实现可以自由选择底层技术，只要对语言层观察到的行为符合本规范。

## 2. 设计原则

### 2.1 Markdown-first

MADL 是嵌入式语言，而不是替代 Markdown 的新容器格式。Markdown 负责文档结构、标题层级、代码块和引用块的承载；MADL 仅在明确定义的位置生效。

该原则的意义在于：

* 语言天然适合写作、审查与协作；
* 说明性文本与执行逻辑可以共处一份文档；
* 解析器可以将 Markdown 层与 DSL 层清晰分离。

### 2.2 显式优于隐式

MADL 不依赖以下隐式机制：

* 隐式变量创建
* 隐式作用域提升
* 隐式闭包捕获
* 隐式共享可变状态
* 隐式 Agent 工具循环控制

任何会影响控制流、数据流、并发边界或宿主交互的行为，都必须由显式语句表达。

### 2.3 数据流优先

MADL 将“值从何处产生、绑定到何处、是否更新已有目标、是否追加到集合”作为一等语义。  
因此语言选择使用统一箭头模型：

* `->`：新绑定
* `=>`：写入已有目标
* `->>`：追加到集合
* `&>` 表示浅合并写入
* `&>>` 表示递归合并写入

该原则确保数据依赖关系在源码中可见，不依赖解释器内部状态推断。

### 2.4 策略静态化

某些语句具有“执行策略”或“解释策略”，例如：

* `.parallel(all)`
* `.eval(lang=python)`
* `.ai(model=gpt-5.4)`
* `.agent(model=gpt-5.4)`
* `.ai-message(role=system)`
* `.http(mode=delegate)`

这些括号中的配置属于**静态策略子句**，必须在解析期确定，不参与变量求值、模板插值或一般表达式运算。

该原则的意义是：

* 解析器可以在构建 AST 时明确语义；
* 静态分析可以检查策略合法性；
* 运行时不必面对“运行时决定语言自身解释方式”的不稳定模型。

### 2.5 值语义优先

MADL 采用值语义作为默认模型：

* `.const` 是深不可变值绑定；
* `.value` 是独占可变值绑定；
* 语言不暴露隐式共享引用语义；
* 语言不支持循环引用或自引用结构。

该原则使：

* 宿主持久化更简单；
* `.parallel` 作用域更安全；
* `.await` 恢复更稳定；
* `.ai`、`.agent`、`.http`、`.callback` 等跨边界数据传递更可序列化。

### 2.6 宿主负责协调，语言负责表达意图

MADL 将某些行为明确视为宿主协调行为，而非解释器内部细节，例如：

* 并发子任务调度
* 外部回调与恢复
* HTTP 委托发送
* 长时间休眠与唤醒
* Agent 工具循环中的工具调用执行
* 外部代码执行服务调用

语言负责表达结构和边界，宿主负责调度与恢复。

## 3. 文档模型

### 3.1 一个文档定义一个 Flow

一个 MADL 文档恰好定义一个 Flow。  
Flow 是 MADL 中的基本可执行单元，也可作为其他 Flow 的调用目标。

### 3.2 文档结构

一个 MADL 文档由两部分组成：

1. **头部定义**：YAML front matter
2. **主体内容**：Markdown 文档，其中部分标题行承载 MADL 指令

### 3.3 头部定义

头部定义必须位于文档开头，格式如下：

```yaml
---
name: path/to/flow
parameters:
  - arg1
  - arg2
description: optional description
---
```

#### 3.3.1 `name`

`name` 是该 Flow 的全局唯一标识。  
它必须是静态字符串，并在仓库或宿主注册表中唯一。

典型形式：

```txt
tools/search_web
demo/ask_user
pipeline/summarize
```

`name` 用于：

* `.call` 目标解析
* `.ai-tool` 工具引用
* 宿主注册表索引
* 日志与执行追踪

#### 3.3.2 `parameters`

`parameters` 定义 Flow 的位置参数列表。  
MADL 的普通函数调用使用位置参数，因此参数顺序具有语义意义。

例如：

```yaml
parameters:
  - query
  - limit
```

表示该 Flow 接收两个位置参数：

1. `query`
2. `limit`

头部参数进入执行环境后表现为只读绑定。

#### 3.3.3 `description`

`description` 是可选元数据。  
语言层不要求其必须存在，但宿主可使用该字段做以下工作：

* 工具描述导出
* 文档展示
* 调试与可观测性
* AI 工具 schema 生成

## 4. Markdown 与指令识别

### 4.1 指令仅存在于标题中

MADL 指令只在 Markdown 标题节点中生效。  
普通段落、列表、表格和正文说明不直接构成指令。

### 4.2 标题内扫描规则

在每个标题文本中，解析器必须从左到右扫描第一个未转义的 `.`，并尝试识别合法关键字。

例如：

```md
# 获取用户资料 .call tools/get_user user_id -> user
```

在该标题中：

* `获取用户资料` 是注释文本
* `.call tools/get_user user_id -> user` 是有效语句

### 4.3 识别步骤

对于任意标题行，解析器应按以下步骤处理：

1. 去除 `#` 前缀及其后一个分隔空格；
2. 在剩余文本中从左到右查找第一个未转义的 `.`；
3. 若其后是合法关键字，则从该位置开始解析 MADL 指令；
4. 该位置之前的内容全部视为注释；
5. 若未找到合法关键字，则该标题行仅作为普通 Markdown 标题存在。

### 4.4 转义

`\.` 表示字面量点号，不触发 DSL 解析。

例如：

```md
# 示例 \.call tools/x
```

不产生任何 MADL 语义。

### 4.5 非关键字形式

若标题中出现 `.foo`，但 `foo` 不是 MADL 关键字，则该标题不构成 MADL 指令。

这一规则保证：

* 普通文档内容不会因相似写法误执行；
* 语言可以在未来扩展关键字，而不破坏旧文档的非语义文本。

## 5. 块级结构与嵌套

### 5.1 标题层级表示嵌套

MADL 通过 Markdown 标题层级表示块级嵌套关系：

* 更深层标题属于最近的更浅层标题的子块
* 同级标题结束前一个同级块
* 更浅层标题结束当前更深层块链

### 5.2 语义嵌套与表示嵌套

MADL 的语义模型是树结构。Markdown 标题只是这种树结构的表示方式。  
由于 Markdown 本身的层级有限，MADL 规范鼓励通过 Flow 拆分与 `.call` 组合实现复杂逻辑，而不是无限制依赖深层嵌套。

### 5.3 子块消费

部分语句会消费其子块内容，例如：

* 条件与迭代语句消费其子语句块
* `.parallel` 消费其并发子流程块
* `.agent` 消费其 `.ai-message` 与 `.ai-tool` 子块
* `.eval`、`.const`、`.value` 可消费紧随的代码块值源

## 7. 表达式模型

### 7.1 表达式使用范围

表达式用于：

* 条件判断：`.if` / `.elif`
* 值源：`.const` / `.value`
* 输出：`.emit`
* 返回：`.return`
* 普通调用参数：`.call`
* HTTP 参数：`.http`
* 等待参数：`.await`
* 睡眠时间：`.sleep`

### 7.2 基本元素

表达式至少应支持：

* 数字字面量
* 字符串字面量
* 布尔字面量
* null
* 对象字面量
* 数组字面量
* 标识符引用
* 属性访问
* 下标访问
* 算术运算
* 比较运算
* 逻辑运算

### 7.3 变量引用

普通变量引用直接使用标识符，例如：

```txt
user
user_id
result
obj.name
arr[0]
```

### 7.4 模板字符串

模板字符串仅在字符串字面量内部使用 `${expr}` 形式插值：

```txt
"hello ${name}"
"${prefix}-${index}"
```

模板字符串不改变一般表达式模型；`${}` 仅是字符串内部插值机制。

### 7.5 运算符

表达式层至少支持以下运算：

#### 算术运算

* `+`
* `-`
* `*`
* `/`
* `%`

#### 比较运算

* `==`
* `!=`
* `>`
* `>=`
* `<`
* `<=`

#### 逻辑运算

* `and`
* `or`
* `not`

### 7.6 语句与表达式分离原则

MADL 明确区分“语句”和“表达式”。  
表达式中不允许出现语句级关键字。特别地：

* 不允许在 `.if` 或 `.elif` 中嵌入 `.call`
* 不允许在表达式中嵌入 `.parallel`、`.agent`、`.await` 等语句

这一原则保证值计算和副作用执行之间的边界稳定。

## 8. 值模型

### 8.1 标准值类型

MADL 标准值包括：

* Number
* String
* Boolean
* Null
* Object
* Array

语言级值模型以 JSON-like 值为基础，但允许表达式层在宿主支持下扩展局部计算能力。

### 8.2 无环值模型

MADL 采用**无环值模型**。  
对象和数组构成的值图必须无环。语言不支持：

* 自引用对象
* 环引用数组
* 通过赋值构造的循环结构

实现若检测到循环值，必须报错。

### 8.3 值与身份

MADL 不暴露对象身份（identity）作为语言可观察语义。  
语言只关心值，而不关心“两个变量是否共享同一底层对象身份”。

这意味着：

* 值相等可以成立；
* 共享引用不作为公开语义；
* 自引用和图结构不作为基础能力。

## 9. 声明与绑定

### 9.1 总体模型

MADL 使用统一的数据绑定模型：

* `->`：创建新绑定
* `=>`：更新已有目标
* `->>`：向数组目标追加

绑定和赋值是语言的一等语义，不通过 `let` / `set` 这类额外机制分裂。

### 9.2 `.const`

`.const` 用于创建不可变绑定。

#### 9.2.1 内联值源

```md
## .const(Type?) expr -> name
## .const(Type?) expr -> [a, b, c]
## .const(Type?) expr -> {name, age}
```

#### 9.2.2 块级值源

````md
## .const(Type?) -> name
```json
...
```
````

````md
## .const(Type?) -> {name, age}
```json
...
```
````

#### 9.2.3 语义

`.const` 的值在语义上必须满足：

1. 与源值隔离；
2. 深不可变；
3. 任意子路径不可写；
4. 可被 `.parallel` 安全读取。

#### 9.2.4 深不可变要求

若 `.const` 的值为对象、数组或嵌套集合结构，语言观察结果必须等价于：

* 深隔离
* 深冻结

实现可以使用：

* 深拷贝
* 写时复制
* 持久化数据结构
* 结构共享 + 冻结

但外部可观察行为必须等价于深不可变值。

#### 9.2.5 类型标注

`Type` 是可选的声明注解，不参与表达式求值。  
其用途包括：

* 静态分析
* 初始化值校验
* 宿主工具导出
* 文档与 IDE 支持

### 9.3 `.value`

`.value` 用于创建可变绑定，或将新值写入已有目标。

#### 9.3.1 初始化绑定（`->`）

```md
## .value(Type?) expr -> name
## .value(Type?) expr -> [a, b, c]
## .value(Type?) expr -> {name, age}
```

#### 9.3.2 块级初始化绑定

````md
## .value(Type?) -> name
```json
...
```
````

````md
## .value(Type?) -> [a, b, c]
```json
...
```
````

#### 9.3.3 更新已有目标（`=>`）

```md
## .value expr => name
## .value expr => obj.key
## .value expr => arr[0]
```

### 9.4 `->`：新绑定

`->` 表示在当前作用域创建一个新的绑定。  
其目标可以是：

* 单个变量名
* 数组解构模式
* 对象解构模式

若当前作用域已存在同名绑定，必须报错。

### 9.5 `=>`：更新已有目标

`=>` 表示将一个新值写入已有目标。  
其目标可以是：

* 单个已有变量
* 已有对象路径
* 已有数组路径

其目标不得是解构模式。

`=>` 只适用于 `.value`，不适用于 `.const`。

### 9.6 `->>`：追加

`->>` 表示向数组目标追加值。

其语义为：

* 若目标不存在，可在当前作用域隐式初始化为空数组后追加；
* 若目标已存在且为数组，则追加；
* 若目标存在但不是数组，则报错。


### 9.7 对象合并写入

MADL 支持将对象值合并写入已有对象目标，以表达对象的增量构造与补丁更新。

#### 9.7.1 语法

对象合并写入仅适用于 `.value`，支持以下两种形式：

#### 内联值源

```md
## .value expr &> target
## .value expr &>> target
```

其中：

* `expr` 是运行时表达式
* `target` 是已有对象目标
* `&>` 表示浅合并写入
* `&>>` 表示递归合并写入

#### 块级值源

````md
## .value &> target
```json
...
```
````

````md
## .value &>> target
```json
...
```
````

当使用块级值源时，紧随其后的代码块内容作为左值来源，其求值结果必须为对象。

#### 9.7.2 总体语义

对象合并写入表示：

1. 求值左侧对象值；
2. 读取右侧 `target` 当前值；
3. 按对应合并规则生成新对象值；
4. 将结果写回 `target`。

该操作属于“对已有目标的增量更新”，不创建新绑定。

#### 9.7.3 目标要求

`target` 必须是**已存在目标**，并且其当前值必须为对象。

允许的目标形式包括：

* 单个已有变量
* 已有对象路径
* 已有数组路径

但目标最终指向的值必须是对象。

例如：

```md
## .value {"email":"alice@example.com"} &> user
## .value {"city":"Singapore"} &>> profile.contact
## .value {"done":true} &> tasks[0]
```

以及：

````md
## .value &> user
```json
{"email":"alice@example.com"}
```
````

若目标不存在，必须报错。
对象合并写入**不执行隐式对象初始化**。

#### 9.7.4 左值要求

左侧值必须为对象。

若使用内联值源，则 `expr` 的求值结果必须为对象。
若使用块级值源，则代码块求值结果必须为对象。

以下写法非法：

```md
## .value 123 &> user
## .value ["age", 18] &>> user
## .value null &> user
```

#### 9.7.5 `&>`：浅合并写入

`&>` 表示浅合并。

其语义为：

* 遍历左值对象的第一层字段；
* 若字段在目标对象中不存在，则新增；
* 若字段在目标对象中已存在，则使用左值字段整体覆盖；
* 不对嵌套对象继续递归合并；
* 若字段值为数组，则整体覆盖，而不是追加或递归处理。

示例：

````md
## .value -> user
```json
{
  "name": "alice",
  "profile": { "age": 18 }
}
```

## .value &> user
```json
{
  "email": "alice@example.com",
  "profile": { "city": "Singapore" }
}
```
````

结果等价于：

```json
{
  "name": "alice",
  "email": "alice@example.com",
  "profile": { "city": "Singapore" }
}
```

#### 9.7.6 `&>>`：递归合并写入

`&>>` 表示递归合并。

其语义为：

* 遍历左值对象的字段；
* 若某字段在目标对象中不存在，则新增；
* 若某字段在左右两侧对应值都为对象，则继续递归合并；
* 除上述情况外，左值字段整体覆盖目标字段；
* 数组不参与递归合并；当字段值为数组时，整体覆盖目标字段；
* 标量值始终整体覆盖。

示例：

````md
## .value -> user
```json
{
  "name": "alice",
  "profile": { "age": 18, "city": "Tokyo" }
}
```

## .value &>> user
```json
{
  "profile": { "city": "Singapore", "email": "a@example.com" }
}
```
````

结果等价于：

```json
{
  "name": "alice",
  "profile": {
    "age": 18,
    "city": "Singapore",
    "email": "a@example.com"
  }
}
```

#### 9.7.7 数组处理规则

无论是 `&>` 还是 `&>>`，数组都不执行元素级合并。

当某字段值为数组时：

* 左值数组整体覆盖目标字段；
* 不追加元素；
* 不按索引递归合并；
* 不执行去重合并。

示例：

````md
## .value -> user
```json
{
  "tags": ["a", "b"]
}
```

## .value &>> user
```json
{
  "tags": ["c"]
}
```
````

结果等价于：

```json
{
  "tags": ["c"]
}
```

#### 9.7.8 与 `=>` 的关系

`=>` 表示对单个已有目标执行普通写入。
`&>` 与 `&>>` 表示对已有对象目标执行对象级合并写入。

因此：

* `=>` 用于替换整个目标值，或写入单个路径位置；
* `&>` 用于对象的一层字段合并；
* `&>>` 用于对象的递归字段合并。

例如：

```md
## .value {"name":"alice"} -> user
## .value 18 => user.age
## .value {"email":"alice@example.com"} &> user
## .value {"profile":{"city":"Singapore"}} &>> user
```

#### 9.7.9 值语义要求

对象合并写入必须保持 MADL 的值语义要求。

语言可观察结果必须等价于：

* 读取目标当前对象值；
* 生成新的合并结果值；
* 将该新值写回目标。

实现可以使用：

* 深拷贝
* 写时复制
* 持久化数据结构
* 结构共享优化

但不得暴露可观察的隐式共享引用。

#### 9.7.10 错误条件

实现必须对以下情况报错：

* 在 `.const` 中使用 `&>` 或 `&>>`
* 目标不存在
* 目标当前值不是对象
* 左值求值结果不是对象
* 合并过程中某目标路径不存在
* 合并目标指向只读绑定
* 合并目标指向头部参数
* 合并目标指向迭代绑定变量
* `.parallel` 子任务中非法写入不可见外层 `.value`


### 9.8 `.value` 的值语义

`.value` 表示可变绑定，但不暴露共享引用语义。  
若一个值被绑定为 `.value`，则对语言观察结果而言，该绑定持有**独占可变值**。

这意味着：

* 对该绑定的更新只影响该绑定；
* 其他绑定不会因隐式共享引用而同步变化；
* 实现可以优化存储，但不得暴露可观察的隐式别名共享。

当 `.const` 的值被用于初始化 `.value` 时，语言观察结果必须等价于获得一份与原值隔离的可变副本。实现可以使用深拷贝或写时复制，但后续对该 `.value` 的修改不得影响原 `.const`。

### 9.9 路径更新语义

当 `=>` 目标是路径时，适用以下规则：

1. 路径起点必须引用一个已存在且可写的 `.value` 绑定；
2. 对象路径上的中间段必须已存在，否则报错；
3. 数组路径的最终索引段允许越界扩容；
4. 数组扩容时，新填充位置的值为 `null`；
5. 仅最终数组索引段允许扩容；若扩容后还需继续向下访问子路径，则按路径不存在报错；
6. 若路径上的任一已有节点类型与后续访问方式不匹配（例如对数组取属性、对对象用数字索引且宿主未定义该行为），必须报错。

### 9.10 设计原则

`->`、`=>`、`->>` 将“新绑定”“更新已有目标”“集合追加”三种意图明确区分。  
该模型比单独 `.set` 语句更统一，也比隐式赋值更适合静态分析。

```md
## .value(Object) {} -> obj
## .value(Any) [] -> variable
```

### 10.2 标准类型名

Version 0.1.0 建议支持的类型名包括：

* `Any`
* `Number`
* `String`
* `Boolean`
* `Null`
* `Object`
* `Array`

### 10.3 语义

类型标注用于声明绑定所期望的值类型。  
实现应在绑定或赋值时执行类型校验：

* 若不匹配，则报错；
* 不应执行隐式类型转换。

### 10.4 `Any`

`Any` 表示不施加具体类型约束，但仍然是显式注解。  
其主要意义在于：

* 表示作者有意放宽该绑定的类型检查；
* 为宿主和工具提供显式“未收窄类型”的信号。

## 11. 解构绑定

### 11.1 数组解构

```md
## .value [1, 2, 3] -> [a, b, c]
```

其语义为：

* 对左侧值求值；
* 按数组顺序解构；
* 在当前作用域分别创建 `a`、`b`、`c`。

若值不是数组或长度不足以满足解构要求，必须报错。

### 11.2 对象解构

```md
## .value {"name":"alice","age":18} -> {name, age}
```

其语义为：

* 左侧值必须为对象；
* 从对象中提取相应字段；
* 在当前作用域创建新绑定。

### 11.3 块级解构

若值源由紧随代码块提供，解构规则保持不变：

````md
## .value -> {name, age}
```json
{"name":"alice","age":18}
```
````

### 11.4 设计原则

解构绑定只属于“新绑定”语义，不属于“更新已有目标”语义。  
因此解构仅与 `->` 搭配，不与 `=>` 搭配。这样可以避免“更新解构目标”带来的语义混乱。

## 12. 作用域

### 12.1 作用域类型

MADL 的作用域由块结构决定，而不是由关键字名决定。  
绑定可处于：

* Flow 顶层作用域
* 条件分支作用域
* 迭代作用域
* 并行子任务作用域

### 12.2 顶层作用域

在文档顶层通过 `.const` / `.value` / `->` 创建的绑定属于 Flow 顶层作用域。

### 12.3 块级作用域

以下结构创建新的局部作用域：

* `.if`
* `.elif`
* `.else`
* `.for`
* `.range`
* `.parallel` 子任务
* `.eval` 执行环境（与 MADL 变量空间隔离，仅通过结果绑定回 MADL）

### 12.4 只读与可写

#### 12.4.1 只读绑定

以下绑定不可被 `=>` 更新：

* 头部 `parameters`
* `.const` 绑定
* 迭代绑定变量
* `.parallel` 子任务中的全局只读可见值

#### 12.4.2 可写绑定

仅 `.value` 绑定可作为 `=>` 的写入目标。

### 12.5 并行中的可见性

`.parallel` 子任务中只能访问：

* 当前绑定变量
* `.const` 绑定
* 头部参数（其本质上也是只读绑定）

子任务不得访问外层 `.value`。  
这是语义规则，不是实现细节。

### 12.6 条件分支中的局部性

`.if` / `.elif` / `.else` 创建的绑定仅在各自分支内部可见，不会自动合流到条件链外部作用域。

因此下述形式必须报“变量未定义”错误：

```md
## .if cond
### .const 1 -> x
## .return x
```

若希望在条件链外使用某个结果，必须先在外层创建一个可写绑定，再在分支内部通过 `=>` 写回。

### 12.7 设计原则

作用域由结构决定，可变性由 `.const/.value` 决定，并行可见性由不可变性决定。  
该三层分工清晰，避免了“global/local + mutability + capture”混杂在一起造成的复杂规则。

## 13. 条件控制

### 13.1 `.if`

```md
## .if expr
```

当 `expr` 为真时，执行其子块。

### 13.2 `.elif`

```md
## .elif expr
```

`.elif` 必须属于某个条件链。当其之前所有 `.if/.elif` 条件均不成立时才求值。

### 13.3 `.else`

```md
## .else
```

`.else` 必须属于某个条件链，且不带表达式。

### 13.4 条件表达式限制

条件表达式不得嵌入语句。  
例如，不允许将 `.call` 直接写入 `.if` 或 `.elif` 的条件中。

### 13.5 设计原则

条件只负责路径选择，不承担副作用。  
将副作用语句排除出条件表达式，可以保证求值阶段不触发隐式执行。

## 14. 迭代 `.for`

### 14.1 语法

```md
## .for items
## .for items as value
## .for items as key value
## .for items as key, value
```

### 14.2 iterable

`items` 是运行时表达式，求值结果必须可迭代。

### 14.3 绑定规则

#### 14.3.1 默认绑定

若省略 `as`：

```md
## .for items
```

默认绑定变量为：

* `index`
* `value`

#### 14.3.2 单变量绑定

```md
## .for items as value
```

仅绑定值。

#### 14.3.3 双变量绑定

```md
## .for items as key value
## .for items as key, value
```

对数组：

* `key` 表示索引
* `value` 表示元素值

对对象：

* `key` 表示键
* `value` 表示值

### 14.4 绑定变量性质

`.for` 产生的绑定变量为只读。

### 14.5 设计原则

`.for` 提供统一的遍历模型，并将“值”和“位置/键”都作为可选绑定暴露给用户。逗号允许但非强制，是为了提高书写宽容度，同时不引入额外语义差异。

## 15. 数值循环 `.range`

### 15.1 语法

```md
## .range 42
## .range 42 as value
## .range 42 as index, value

## .range start end
## .range start end as value
## .range start end as index, value

## .range start end step
## .range start end step as value
## .range start end step as index, value
```

### 15.2 基本语义

#### 15.2.1 单参数

`.range 42` 等价于：

* `start = 0`
* `end = 42`
* `step = 1`

#### 15.2.2 双参数

`.range start end` 表示：

* 起点 `start`
* 终点 `end`
* 步长默认 `1`

#### 15.2.3 三参数

`.range start end step` 表示完整形式。

### 15.3 绑定规则

与 `.for` 一致：

* 默认：`index`, `value`
* 单变量：只绑定值
* 双变量：绑定迭代序号与当前值

### 15.4 绑定变量性质

`.range` 绑定变量为只读。

### 15.5 设计原则

`.range` 与 `.for` 共享相同的绑定模式，以降低认知成本并提高语法一致性。

## 16. 并行 `.parallel`

### 16.1 语法

```md
## .parallel(all) items
## .parallel(all) items as value
## .parallel(all) items as index value
## .parallel(all) items as index, value -> results

## .parallel(race) items as value -> winner
```

### 16.2 策略子句

`.parallel` 的括号内容是静态策略子句。  
当前标准策略包括：

* `all`
* `race`

其值必须为静态字面量，不允许变量解析。

### 16.3 iterable 与绑定

`items` 是运行时表达式。绑定规则与 `.for` 相同。

### 16.4 子块

`.parallel` 必须有子块。每个 iterable 元素将对该子块逻辑生成一个独立子执行单元。

### 16.5 宿主委托并发模型

`.parallel` 不规定解释器内部线程模型。  
其语义是：解析器与运行时将并行任务模板委托给宿主，由宿主并发创建多个子实例执行子块。

宿主接收的信息至少包括：

* 策略
* iterable 项
* 当前可见 `.const` 绑定
* 当前绑定变量值
* 子块执行体
* 当前 VM 恢复元数据

### 16.6 子实例作用域

每个并发子实例拥有：

* 独立局部作用域
* 当前绑定变量
* 对 `.const` 值的只读访问
* 对 `.value` 的隔离访问（实际上禁止从外层读取）

### 16.7 变量可见性规则

在 `.parallel` 子块中，允许访问：

* 当前绑定变量
* `.const` 绑定
* 头部只读参数

禁止访问：

* 外层 `.value`

这是强制语义，不允许实现放宽为闭包捕获。

### 16.8 返回值收集

#### 16.8.1 `all`

在 `.parallel(all) ... -> results` 中：

* 所有子实例运行完成；
* 子块中的 `.return` 值按输入顺序聚合为数组；
* 绑定给 `results`。

若子块未显式 `.return`，该子实例结果为 `null`。

#### 16.8.2 `race`

在 `.parallel(race) ... -> winner` 中：

* 最先成功完成并返回结果的子实例获胜；
* 其余实例可由宿主尽力取消，或在结果层面被忽略；
* 获胜值绑定给 `winner`。

### 16.9 `.emit` 与并行

`.emit` 是宿主函数。  
并行中的 `.emit` 输出顺序按实际执行顺序，不保证按输入顺序排列。  
`.emit` 不参与 `->` 聚合结果。

### 16.10 错误传播

* 在 `all` 策略下，任一子实例失败应导致整个 `.parallel(all)` 失败。
* 在 `race` 策略下，若所有子实例都失败，则整个 `.parallel(race)` 失败。

### 16.11 `race` 中的副作用说明

`.parallel(race)` 的“落败”子实例不保证已经在宿主层面被立即中止。  
因此：

* 胜负只决定最终返回值绑定；
* 不保证落败子实例的副作用一定回滚；
* 宿主可以尽力取消落败实例，但语言不以“副作用已取消”作为可观察保证。

### 16.12 设计原则

`.parallel` 被设计为结构化并发意图，而不是共享内存线程语法。  
通过“静态策略 + 显式绑定 + 禁止捕获 + 宿主委托”，语言能够在不同宿主下实现一致的并发语义。

## 17. 调用 `.call`

### 17.1 语法

```md
## .call path/to/flow arg1 arg2
## .call path/to/flow arg1 arg2 -> result
## .call path/to/flow arg1 arg2 ->> results
```

### 17.2 目标路径

`path/to/flow` 必须是静态路径字面量，对应某个已注册 Flow 的 `name`。  
它不参与变量解析，不允许模板插值。

### 17.3 参数

`.call` 使用位置参数。参数按目标 Flow 头部 `parameters` 顺序绑定。  
参数值属于运行时表达式，应在调用前求值。

### 17.4 返回值

被调用 Flow 的 `.return` 值作为 `.call` 的结果。  
随后可使用 `->` 或 `->>` 进行绑定。

### 17.5 失败模型

`.call` 应优先返回结构化结果。  
仅当目标 Flow 不存在、参数数量不匹配、运行时无法建立调用或目标执行发生系统级失败时，`.call` 语句本身失败。

业务层失败应由被调用 Flow 通过结构化返回值表达，而不是依赖语言级异常。

### 17.6 设计原则

`.call` 的目标静态化，是为了让调用图可分析、可校验、可导出。  
使用位置参数是为了保持标题语句紧凑。

---

## 18. 输出 `.emit`

### 18.1 语法

```md
## .emit expr
```

### 18.2 语义

`.emit` 将 `expr` 的值交给宿主。  
它不产生可绑定结果，不改变 Flow 的返回值，不参与 `->` 数据流。

### 18.3 设计原则

`.emit` 是宿主副作用出口，而不是数据绑定语句。  
语言通过将 `.emit` 与 `.return` 分离，明确区分：

* 内部计算结果
* 对外部世界的通知/日志/消息

---

## 19. 返回 `.return`

### 19.1 语法

```md
## .return expr
```

### 19.2 顶层语义

在普通 Flow 顶层执行中，`.return` 立即终止当前 Flow，并返回 `expr` 的值。

### 19.3 并行子实例语义

在 `.parallel` 子块中，`.return` 终止当前子实例，并将其值交给 `.parallel` 聚合器。

### 19.4 设计原则

`.return` 始终结束“当前可返回执行单元”。  
顶层是 Flow，并行内部是子实例。该统一原则使解释器和运行时更易实现。

## 20. 外部求值 `.eval`

### 20.1 语法

```md
## .eval(lang=python) -> result
```

紧随其后的代码块为执行体。

### 20.2 策略子句

`lang=python` 属于静态策略子句，不参与变量求值。

### 20.3 执行模型

* `.eval` 读取紧随代码块作为执行体；
* 当前运行时不要求在 VM 内直接执行该代码；
* 宿主可将其委托给外部代码执行服务；
* 当前可见值环境可被序列化后以只读方式传入外部执行环境；
* 外部执行结果通过 `->` 或 `->>` 进入 MADL。

### 20.4 语义边界

`.eval` 是一种外部委托求值能力，而不是在 MADL VM 内嵌入另一种语言的本地执行环境。  
实现可以通过 HTTP、RPC、子进程或其他宿主机制完成委托。

### 20.5 设计原则

`.eval` 用于桥接外部求值能力，但不改变 MADL 的核心变量和作用域语义。  
它是“外部求值并返回结果”的结构，不是“在 MADL 内嵌入另一种语言作用域”。

## 21. 单次 AI 调用 `.ai`

### 21.1 语法

```md
## .ai(model=gpt-5.4) messages tools? -> result
```

或以块级值源/宿主约定形式提供输入。

### 21.2 策略子句

当前标准策略至少包括：

* `model=<literal>`

`model` 必须为静态字面量，不参与变量解析。

### 21.3 语义

`.ai` 表示一次单轮模型调用，而不是自动工具循环。  
其输入至少应包括：

* 模型配置
* 消息列表
* 可选工具列表

模型执行一次后返回结果。

### 21.4 返回值

`.ai` 的返回值由宿主定义，但应为结构化值，至少能够表达：

* 模型直接返回的最终内容；或
* 模型请求调用某个工具的意图

### 21.5 设计原则

`.ai` 是基础模型调用原语。  
它只负责“一次模型推理”，不承担循环控制、工具自动执行与轮次状态管理。

## 22. Agent 调用 `.agent`

### 22.1 语法

```md
## .agent(model=gpt-5.4) -> result
```

`.agent` 必须有子块，用于构建 Agent 上下文。

### 22.2 策略子句

当前标准策略至少包括：

* `model=<literal>`

`model` 必须为静态字面量，不参与变量解析。

### 22.3 Agent 上下文

执行 `.agent` 时，运行时应建立一个 Agent 上下文，至少包含：

* 模型配置
* 消息列表
* 工具列表
* 执行轮次状态

### 22.4 执行模型

`.agent` 是语言级自动工具循环原语：

1. 执行 `.agent` 子树，收集消息与工具；
2. 进行一次 `.ai` 等价的模型调用；
3. 若模型直接返回最终结果，则结束；
4. 若模型请求调用工具，则执行工具；
5. 将工具结果追加回 Agent 上下文；
6. 重复调用模型，直到得到最终结果或达到宿主上限。

### 22.5 结果绑定

`.agent` 的最终结果通过 `-> result` 绑定到目标变量。

### 22.6 设计原则

`.agent` 将“消息构建”和“工具循环执行”合并为单一语言原语，使用户不必手写重复的 agentic loop，同时保证 Agent 调用边界和工具能力边界显式可见。

---

## 23. AI 消息 `.ai-message`

### 23.1 语法

```md
## .ai-message(role=system)
> ...
```

### 23.2 上下文要求

`.ai-message` 只能出现在 `.agent` 子树中。  
若出现在 `.agent` 之外，属于语义错误。

### 23.3 角色

`role` 是静态策略子句。建议支持：

* `system`
* `user`
* `assistant`
* `developer`

### 23.4 内容来源

`.ai-message` 的消息体由其附属正文提供。推荐使用引用块表示正文。正文允许模板字符串插值。

### 23.5 执行语义

当 `.ai-message` 被执行时：

* 其正文在当前作用域渲染；
* 形成一条消息；
* 追加到当前 `.agent` 上下文消息列表。

### 23.6 控制流中的消息构建

`.ai-message` 可出现在 `.if`、`.for`、`.range` 等子块中。其是否追加及追加顺序由控制流执行结果决定。

### 23.7 设计原则

`.ai-message` 是“Agent 上下文构造语句”，而不是“即时模型调用语句”。  
该设计使消息可以由普通控制流动态构造，并在 `.agent` 执行时统一生效。

---

## 24. AI 工具 `.ai-tool`

### 24.1 语法

```md
## .ai-tool path/to/function
```

### 24.2 上下文要求

`.ai-tool` 只能出现在 `.agent` 子树中。  
若出现在 `.agent` 之外，属于语义错误。

### 24.3 路径要求

工具路径必须是静态字面量，对应已注册 Flow 的 `name`。  
它不参与变量解析。

### 24.4 注册语义

执行 `.ai-tool` 时，运行时应：

1. 查找该 Flow；
2. 读取其参数签名和元数据；
3. 将其转换为当前 `.agent` 上下文中的可调用工具。

### 24.5 工具调用执行

当模型在 `.agent` 自动循环中请求调用某工具时，运行时必须：

1. 验证该工具已注册；
2. 校验调用参数；
3. 执行对应 Flow；
4. 获取其 `.return` 值；
5. 将该结果以 AI 可理解形式追加回当前上下文；
6. 继续下一轮模型调用。

### 24.6 失败模型

`.ai-tool` 的业务层失败应优先通过工具返回值表达。  
仅当工具未注册、参数校验无法建立、调用执行发生系统级失败或宿主无法继续 Agent 循环时，`.agent` 语句本身失败。

### 24.7 设计原则

`.ai-tool` 的职责是暴露受控工具边界，而不是开放任意动态调用。  
路径静态化和注册显式化，是为了保证 Agent 能力边界可审计、可校验、可限制。

---

## 25. 宿主协调总览

MADL 支持显式宿主协调语句，用于表达：

* 回调令牌创建
* 挂起与恢复
* 宿主委托 HTTP
* 长时间休眠

这些语句共同的设计原则是：

* 等待行为显式表达；
* VM 可以被持久化；
* 恢复点明确；
* 恢复数据如何进入变量系统明确。

---

## 26. 回调 `.callback`

### 26.1 语法

```md
## .callback desc -> token
```

### 26.2 语义

`.callback` 要求宿主生成一个回调 token。执行时：

1. 对 `desc` 求值；
2. 宿主创建 token；
3. 宿主记录当前 VM、执行位置与元数据；
4. 将 token 绑定给 `token`。

### 26.3 token 的用途

token 可发送给外部服务、人工流程或异步系统，用于在未来将恢复事件关联回当前 VM。

### 26.4 设计原则

`.callback` 只负责“创建外部恢复入口”，不负责“阻塞等待”。  
等待行为由 `.await` 表达，以保持语义分层清晰。

---

## 27. 等待 `.await`

### 27.1 语法

```md
## .await token -> data
```

### 27.2 语义

`.await` 表示当前 VM 暂停执行并等待与 `token` 对应的恢复事件。

执行时：

1. 对 `token` 求值；
2. 宿主验证 token；
3. 当前 VM 进入等待状态；
4. 宿主可持久化 VM；
5. 外部恢复事件到达后，宿主恢复 VM；
6. 将恢复数据绑定到 `data`；
7. 从 `.await` 之后继续执行。

### 27.3 设计原则

`.await` 是语言级暂停点，不是解释器内部阻塞技巧。  
该设计保证 MADL 可以原生支持长生命周期与人工参与工作流。

---

## 28. HTTP `.http`

### 28.1 语法

```md
## .http(mode=direct) GET url headers -> status, headers_out, data
## .http(mode=delegate) POST url headers -> status, headers_out, data
```

### 28.2 策略子句

`mode` 属于静态策略子句。标准模式包括：

* `direct`
* `delegate`

### 28.3 参数

`.http` 的一般形式为：

```txt
.http(mode=<mode>) <method> <url> <headers> -> <status>, <headers_out>, <data>
```

其中：

* `method` 是运行时表达式，通常为方法名；
* `url` 是运行时表达式；
* `headers` 是运行时表达式，通常为对象。

### 28.4 返回值

`.http` 返回三元结果：

1. `status`
2. `headers_out`
3. `data`

### 28.5 HTTP 状态语义

* 非 2xx HTTP 状态码**不构成语言级失败**，仅作为普通 `status` 返回值；
* 业务逻辑应显式检查 `status` 决定后续流程；
* 该原则同样适用于其他外部能力调用：优先返回结构化结果，而不是将业务失败升级为语言级异常。

### 28.6 传输层失败

若发生以下情况：

* DNS 解析失败
* 连接失败
* TLS 握手失败
* 请求超时
* 宿主未获得任何有效 HTTP 响应

则：

* `status = 0`
* `headers_out = {}`
* `data` 应为结构化错误对象，至少包含可读描述

例如：

```json
{"error":"timeout","message":"request timed out"}
```

### 28.7 `mode=direct`

在 direct 模式下：

* 由当前运行时直接发送 HTTP 请求；
* 请求完成后继续执行；
* 不要求进入宿主持久化等待。

### 28.8 `mode=delegate`

在 delegate 模式下：

* 由宿主接管 HTTP 请求；
* 当前 VM 可被挂起和持久化；
* 宿主在拿到 HTTP 响应后恢复 VM；
* 响应值绑定到返回变量。

### 28.9 设计原则

`.http` 的 mode 区分“谁负责执行网络行为”，而不是“请求内容是什么”。  
通过 mode 静态化，MADL 可以统一表达同步请求与宿主托管请求。

---

## 29. 运行时与 VM 模型

### 29.1 VM

运行 MADL Flow 时，运行时应维护一个可恢复 VM，至少包含：

* 当前 Flow 标识
* 参数绑定
* 当前作用域栈
* 当前执行位置
* 并发与等待状态
* 宿主恢复元数据

### 29.2 挂起点

以下语句是标准挂起点或可挂起点：

* `.await`
* `.http(mode=delegate)`
* `.sleep`
* `.parallel`（由宿主调度时）
* `.agent`（若工具循环或模型调用由宿主托管）
* `.eval`（若外部求值由宿主托管）

### 29.3 恢复要求

宿主恢复 VM 后，语言可观察结果必须与“未中断但语义等价地继续执行”一致。

### 29.4 设计原则

MADL 将中断与恢复视为标准执行路径，而非异常情况。  
这使语言天然适合：

* Agent 长流程
* Human-in-the-loop
* 异步外部任务
* 分布式执行

---

## 30. 休眠 `.sleep`

### 30.1 语法

```md
## .sleep unix_ms
```

### 30.2 语义

`.sleep` 表示当前 VM 暂停执行，直到宿主决定恢复。  
`unix_ms` 是目标恢复时间或宿主约定的唤醒时刻表达。

执行时：

1. 对 `unix_ms` 求值；
2. 将其交给宿主；
3. 宿主选择：
   * 内存内短时等待后继续；
   * 持久化 VM，待时间到后恢复；
4. 到达恢复时机后继续执行。

### 30.3 设计原则

`.sleep` 表示“计划恢复时间”，而不是“解释器必须阻塞某个毫秒数”。  
将恢复控制交给宿主，才能支持长期等待与跨进程恢复。

## 31. 错误条件

实现必须对以下情况报错：

### 31.1 绑定与作用域错误

* 引用未定义变量
* 使用 `->` 绑定已在当前作用域存在的名称
* 使用 `=>` 更新不存在的目标
* 更新 `.const`
* 更新头部参数
* 更新迭代绑定变量
* `.parallel` 内访问外层 `.value`
* 条件分支内声明的名称被错误地在分支外引用

### 31.2 类型错误

* 声明类型标注与值不匹配
* `->>` 目标不是数组
* 解构模式与值类型不匹配

### 31.3 路径更新错误

* 对象路径中间段不存在
* 在非最终数组索引段尝试依赖扩容继续向下访问
* 路径节点类型与后续访问方式不匹配

### 31.4 结构错误

* `.parallel` 无子块
* `.elif` / `.else` 不构成合法条件链
* `.ai-message` 或 `.ai-tool` 出现在 `.agent` 外部
* `.const` 使用 `=>`

### 31.5 值模型错误

* 检测到循环引用或自引用结构

### 31.6 策略子句错误

* 策略子句中出现变量解析、模板插值或一般表达式

### 31.7 调用错误

* `.call` 或 `.ai-tool` 引用不存在的 Flow
* 调用参数数量与目标签名不匹配

---

## 32. 一致性要求

符合本规范的实现必须满足以下一致性要求：

1. 同一源码在相同输入下，控制流路径必须一致；
2. `.parallel(all)` 聚合结果顺序必须按输入顺序；
3. `.parallel` 不得允许闭包捕获外层 `.value`；
4. `.const` 必须表现为深不可变值；
5. `.value` 不得暴露可观察的隐式共享引用；
6. `.agent` 必须按自动工具循环模型工作；
7. `.ai` 必须表现为单次模型调用，而不是自动循环；
8. `.await` 恢复后必须从明确暂停点继续；
9. `.http(mode=delegate)` 必须允许宿主挂起并恢复 VM；
10. `.sleep` 必须允许宿主决定内存等待或持久化恢复；
11. 非 2xx HTTP 响应不得被自动升级为语言级失败；
12. 传输层失败必须可通过 `status = 0` 被语言观察到。

## 33. 示例

### 33.1 基础绑定

```md
---
name: demo/bindings
parameters:
  - user_name
---

## .const(String) "hello ${user_name}" -> greeting
## .value(Number) 1 -> counter
## .value 2 => counter
## .return {"greeting": greeting, "counter": counter}
```

### 33.2 解构与块级值源

````md
---
name: demo/destructure
parameters: []
---

## .const -> {name, age}
```json
{"name":"alice","age":18}
```

## .value -> [a, b, c]
```json
[1, 2, 3]
```

## .return {"name": name, "age": age, "sum": a + b + c}
````

### 33.3 条件链外写回

```md
---
name: demo/if-writeback
parameters:
  - cond
---

## .value(Any) null -> x

## .if cond
### .value 1 => x
## .else
### .value 2 => x

## .return x
```

### 33.4 并行

```md
---
name: demo/parallel
parameters:
  - items
---

## .const(String) "item" -> prefix

## .parallel(all) items as index, value -> results
### .const {"label": "${prefix}-${index}", "value": value} -> item_result
### .return item_result

## .return results
```

### 33.5 Human-in-the-loop

```md
---
name: demo/approval
parameters:
  - order_id
---

## .callback {"type":"approval","order_id":order_id} -> token
## .emit {"approval_token": token, "order_id": order_id}
## .await token -> approval

## .if approval.status == "approved"
### .return {"ok": true, "order_id": order_id}
## .else
### .return {"ok": false, "reason": approval.reason}
```

### 33.6 单次 AI 调用

```md
---
name: demo/ai_once
parameters:
  - messages
---

## .ai(model=gpt-5.4) messages -> result
## .return result
```

### 33.7 Agent with tools

```md
---
name: demo/ask
parameters:
  - user_query
---

## .agent(model=gpt-5.4) -> result

### .ai-message(role=system)
> You are a helpful assistant.
> Use tools only when necessary.

### .ai-tool tools/search_web
### .ai-tool tools/fetch_page

### .ai-message(role=user)
> ${user_query}

## .return result
```

### 33.8 HTTP 状态处理

```md
---
name: demo/http_status
parameters:
  - url
---

## .http(mode=direct) "GET" url {} -> status, headers_out, data

## .if status == 200
### .return {"ok": true, "data": data}
## .else
### .return {"ok": false, "status": status, "data": data}
```
