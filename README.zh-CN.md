# MADL

> Markdown Agent Description Language
>
> 一种嵌入在 Markdown 文档中的、面向 Agent 与长流程自动化的可执行语言。

MADL 试图把**文档**与**程序**合并为同一个产物：
同一份文件既可以被人阅读、审查、讨论，也可以被运行时解析、调度与执行。

它不是一个提示词格式，也不是一个工作流配置文件的变体。
MADL 的目标，是为 **Agent Flow、工具调用、宿主协调、长生命周期任务、人工参与流程** 提供一套语义清晰、可静态分析、可恢复执行的语言模型。

---

## 为什么是 MADL

今天的大多数 Agent 系统都建立在脆弱的运行时约定之上：

- 提示词负责控制流程
- JSON 负责传递半结构化状态
- 宿主代码负责拼接工具调用
- 人工回调、长时间等待、异步恢复往往只是框架层补丁

这样可以快速做出原型，但很难形成**稳定、可审查、可组合、可恢复**的系统。

MADL 想解决的是更底层的问题：

- 如何把 **控制流** 和 **数据流** 写得显式
- 如何让 **等待、恢复、并发、宿主委托** 成为语言级语义
- 如何让 **AI 调用与工具循环** 成为可定义、可限制、可分析的结构
- 如何让一份文档在保持可读性的同时，又具有程序的确定性

## 核心理念

### Markdown-first

MADL 不是替代 Markdown 的新容器格式。
它是一种嵌入在 Markdown 中的语言：文档结构继续由 Markdown 承载，执行语义只在明确的位置生效。

### 显式优于隐式

MADL 不依赖隐式变量创建、隐式作用域提升、隐式闭包捕获、隐式共享可变状态。
任何影响控制流、数据流或宿主交互边界的行为，都应由源码显式表达。

### 值语义优先

MADL 以值语义为默认模型：

- `.const` 绑定深不可变值
- `.value` 绑定独占可变值
- 不暴露可观察的共享引用语义
- 并发与恢复以可序列化、可隔离的数据为基础

### 宿主负责协调，语言负责表达意图

MADL 不试图把所有调度逻辑都塞进解释器内部。
它把并发调度、长时间等待、回调恢复、宿主委托 HTTP、Agent 工具执行等能力，明确建模为语言边界与宿主责任的协作关系。

## 一个简单例子

```md
---
name: demo/approval
parameters:
  - order_id
---

## .callback {"type":"approval","order_id":order_id} -> token
## .emit {"approval_token": token, "order_id": order_id}
## .await token -> approval

## .if approval.status == "approved"
### .return {"ok": true, "order_id": order_id}
## .else
### .return {"ok": false, "reason": approval.reason}
```

这个例子展示了 MADL 最核心的方向：
**等待与恢复不是异常情况，而是语言的一等执行路径。**

---

## 它想支持什么

MADL 面向的是这类流程：

- Agent 驱动的多步任务
- 带工具调用的模型交互
- Human-in-the-loop 审批与介入
- 跨进程、跨时间的长生命周期工作流
- 需要持久化 VM 状态并在未来恢复的执行模型
- 需要在“文档可读性”和“程序可执行性”之间取得平衡的系统

---

## 与常见工作流 DSL 的不同点

MADL 不是把 YAML、JSON 或可视化节点图换一种语法皮肤。
它有几条非常明确的语言立场：

- **指令嵌入在 Markdown 中，而不是与文档分离**
- **控制流与数据流是源码的一部分，不依赖运行时约定猜测**
- **等待、恢复、宿主委托、Agent 循环属于语言核心，而不是外部框架插件**
- **静态策略子句在解析期确定，不允许运行时再决定语言自身解释方式**
- **并发、恢复与工具边界以值语义和隔离语义为前提**

---

## 当前状态

MADL 目前仍处于**语言设计阶段**。

当前仓库重点是：

- 语言规范
- 核心语义模型
- 运行时边界设计
- 未来实现的约束与方向

目前**还没有完整运行时实现**。

这意味着：

- 仓库当前主要是提案与规范，而不是可直接投入生产的执行引擎
- 设计仍可能继续调整，但核心方向已经比较明确
- 现在最重要的是先把语义打磨正确，而不是过早绑定某个具体实现细节

---

## 设计目标

MADL 希望最终具备这些特性：

- 可读：适合写作、审查、协作与讨论
- 可执行：同一份文档可直接作为 Flow 定义
- 可分析：大部分结构、作用域、绑定和引用错误可在执行前发现
- 可恢复：支持 `.await`、`.sleep`、宿主恢复与长流程任务
- 可组合：一个文档定义一个 Flow，通过调用和工具机制组合更复杂系统
- 可约束：AI、工具、HTTP、回调、并发边界都能被显式表达与控制

## 这个项目适合谁

如果你对下面这些问题感兴趣，MADL 可能值得关注：

- Agent workflow 应该是一种语言，还是只是框架 API？
- 长生命周期任务的暂停与恢复，应该如何进入语言语义？
- 文档与程序是否可以共享同一个源文件？
- LLM 工具调用循环是否应该是语言原语，而不是 prompt 约定？
- 如何在保持表达力的同时，避免工作流系统退化成“到处都是隐式状态和魔法约定”？

## 路线图

接下来预计会逐步推进：

- 继续打磨规范与语义闭合性
- 明确运行时与宿主接口边界
- 设计 AST / 检查器 / VM 模型
- 实现最小可运行原型
- 补充示例、测试与错误模型

---

## 参与方式

MADL 目前非常适合讨论这些方向：

- 语言语义是否自洽
- 作用域、绑定、恢复与并发模型是否合理
- `.agent` / `.ai` / `.http` / `.eval` 等原语是否分层正确
- 哪些设计值得保守，哪些地方应该进一步简化

欢迎 issue、讨论、设计反馈与批评。

---

## 状态声明

MADL 目前还是一个**语言提案**。

它还没有完整实现，很多价值也尚未被运行时代码证明。
但它的目标很明确：

> 不是再发明一种工作流配置文件，
> 而是为 Agent 与长流程自动化认真设计一门语言。
